# GPT-5.5時代のプロンプト設計は「命令」から「契約」へ

OpenAIの Prompt Guidance と Using GPT-5.5 Guide を読むと、プロンプト設計の考え方がかなり変わってきている。

一言でいうと、

> 細かく手順を縛るより、目的・制約・成功条件を定義して任せる

という方向。

GPT-4時代のプロンプトは「AIをどう動かすか」を細かく指定するものだった。
GPT-5.5時代のプロンプトは「AIに何を達成してほしいか」を明確にするものになっている。

---

## 1. 一番大きい変化は Outcome-first

以前のプロンプトは、こうなりがちだった。

```text
1. まずAを確認
2. 次にBを調査
3. その後Cを比較
4. 最後にJSONで出力
```

でも、GPT-5.5ではこういう書き方のほうが安定しやすい。

```text
目的:
競合比較を行い、意思決定できる結論を出すこと。

制約:
- 信頼できる情報のみ使う
- 不明点は不明と書く
- 結論と根拠を分ける
- 最後に推奨アクションを出す
```

つまり、「考え方の手順」を指定するより、「達成条件」を渡す。

---

## 2. 長いプロンプトは、むしろ邪魔になることがある

GPT-5.5は instruction following がかなり強い。

そのため、古いプロンプトをそのまま積み増すと、逆に悪くなる。

起きやすい問題はこれ。

- 指示が多すぎて過剰拘束になる
- 古い指示と新しい指示が衝突する
- どうでもいい確認質問が増える
- 出力が丁寧すぎて長くなる
- 本来不要な調査やツール利用が増える

これからのプロンプト改善は「足す」より「削る」が重要。

最初から巨大なプロンプトを書くのではなく、最小のプロンプトから始めて、失敗したところだけ制約を足す。

---

## 3. AIに「手順」ではなく「仕事」を渡す

GPT-4時代は、AIを暴走させないために細かく管理する感覚が強かった。

GPT-5.5では、もう少し仕事を委譲できる。

例えば以前なら、

```text
検索して
比較して
不足を確認して
最後に要約して
```

と書いていたところを、

```text
必要な調査を行い、比較と検証を含めて、
意思決定できる形でまとめてください。
```

で動かせる場面が増える。

ここはかなり Agent 的になっている。

---

## 4. ただし、自由にしすぎると広がりすぎる

GPT-5.5は強いが、万能に放置していいわけではない。

むしろ自由度が高いぶん、制約が重要になる。

良いプロンプトに必要なのは、だいたいこの5つ。

```text
Goal:
何を達成するか

Context:
背景・前提・使ってよい情報

Constraints:
やってはいけないこと、守るべき条件

Output:
欲しい形式・粒度・長さ

Evaluation:
良い回答の条件
```

特に重要なのは Constraints。

「何をするか」だけでなく、「どこまでやるか」「何をしないか」を書く。

---

## 5. reasoning は毎回書かせなくていい

以前は「step by stepで考えて」が万能のように扱われていた。

でも、GPT-5.5では毎回それをやると逆効果になることがある。

- 出力が冗長になる
- 考えすぎる
- 余計な仮定が増える
- シンプルな作業まで重くなる

通常タスクでは concise に。
難しい判断、設計、検証が必要なときだけ reasoning を厚くする。

---

## 6. APIでは prompt だけでなく parameter も使う

GPT-5.5では、プロンプトだけですべてを制御しようとしないほうがいい。

たとえば、

- reasoning_effort: 思考の深さ
- verbosity: 出力の密度
- tool behavior: ツール利用の積極性

のような設定で調整する。

つまり、プロンプトは「仕事の契約」。
パラメータは「働き方の設定」。

この2つを分けて考えるのが実務ではかなり大事。

---

## まとめ

GPT-5.5時代のプロンプト設計は、こう変わる。

- 手順指定より Outcome-first
- 長いプロンプトより小さく始める
- AIを細かく操作するより仕事を委譲する
- 自由度を上げるぶん制約を明確にする
- reasoning は必要なときだけ厚くする
- prompt と parameter を分けて設計する

要するに、これからのPrompt Engineeringは「命令を書く技術」ではなく、

> AIに仕事を渡すための契約設計

に近づいている。
